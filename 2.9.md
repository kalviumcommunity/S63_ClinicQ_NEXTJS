TypeScript & ESLint Configuration

Now that your project is initialized, it’s time to bring in code quality and consistency. In this task, you’ll configure strict TypeScript settings, set up ESLint + Prettier, and enforce them through pre-commit hooks — ensuring your team writes clean, consistent, and bug-free code throughout the sprint.

Note: These are general submission guidelines. You may customize your prompts, PR titles, and README reflections to fit your allocated problem statement and team context.

What You’ll Do
Enable Strict TypeScript Configuration

Open your tsconfig.json and set:

{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true
  }
}
These rules help catch potential errors early — preventing undefined types, unused code, or casing mismatches.

Set Up ESLint + Prettier

Install the required dependencies:

npm install eslint prettier eslint-config-next eslint-plugin-prettier eslint-config-prettier --save-dev
Create or update .eslintrc.json:

{
  "extends": ["next/core-web-vitals", "plugin:prettier/recommended"],
  "rules": {
    "no-console": "warn",
    "semi": ["error", "always"],
    "quotes": ["error", "double"]
  }
}
Create a .prettierrc file for consistent formatting:

{
  "singleQuote": false,
  "semi": true,
  "tabWidth": 2,
  "trailingComma": "es5"
}
Add Pre-Commit Hooks

Use Husky and lint-staged to enforce checks before commits:

npx husky-init && npm install
npm install lint-staged --save-dev
Update package.json:

"lint-staged": {
  "*.{ts,tsx,js,jsx}": ["eslint --fix", "prettier --write"]
}
Test by committing code that breaks a lint rule — it should fail until fixed.
